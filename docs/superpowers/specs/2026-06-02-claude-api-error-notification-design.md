# Notify on Claude API-error turn death

**Date:** 2026-06-02
**Status:** Approved (brainstorming → ready for implementation plan)

## Problem

When a Claude Code session's turn ends because of an API error — e.g. the
`API Error: Server is temporarily limiting requests (not your usage limit) ·
Rate limited` message observed in the "Securus Interim Pull" worktree — the
agent thread silently dies. TBD shows no notification, so the user has no idea
the session stalled until they happen to look at the terminal. By the time this
error surfaces, Claude Code has already auto-retried ~10× (configurable via
`CLAUDE_CODE_MAX_RETRIES`), so it is a genuine "this turn is dead" signal, not a
transient flap.

## Root cause

TBD's Claude hook overlay registers a `Stop` hook (which drives the
`response_complete` notification), but Claude Code's `Stop` hook fires **only on
normal completion**. When a turn is killed by an API error, `Stop` never fires —
hence the silence.

## Key finding

Claude Code (verified against the installed binary, **v2.1.160**) has a
dedicated **`StopFailure`** hook that fires precisely when a turn ends due to an
API error. It is distinct from `Stop`. On stdin it receives:

```json
{
  "session_id": "…",
  "transcript_path": "/…/transcript.jsonl",
  "cwd": "/…",
  "hook_event_name": "StopFailure",
  "error_type": "rate_limit | server_error | authentication_failed | billing_error | oauth_org_not_allowed | invalid_request | model_not_found | max_output_tokens | unknown"
}
```

It supports a matcher on `error_type` and is **non-blocking** (we can react, but
cannot retry the turn). The error is also written to the transcript JSONL
(`isApiErrorMessage: true`, `apiErrorStatus: 429`, `error: "rate_limit"`), which
confirms a transcript-scanning path exists — but the hook is the cleaner,
push-based mechanism and needs no polling. We do **not** build a transcript
scanner / `asyncRewake`.

## Design

Register a `StopFailure` event in the TBD-owned Claude overlay generated by
`Sources/TBDDaemon/Hooks/ClaudeHookOverlay.swift`, alongside the existing
`SessionStart` / `Stop` / `PreToolUse` / `PostToolUse` events.

No matcher → it catches **all** `error_type`s. (A silently dead thread is bad
regardless of cause.)

### Command (Option A — inline `jq`)

The hook command mirrors the existing `Stop` hook's inline-`jq` pattern
(`stopCommand`, which extracts `last_assistant_message`). It extracts
`error_type` and shells out to the existing `tbd notify` CLI:

```sh
TYPE=$(jq -r '.error_type // "unknown"' 2>/dev/null); tbd notify --type error --message "Claude stopped: API error ($TYPE)" 2>/dev/null || true
```

It is added as a new `static let stopFailureCommand` constant and wired into
`generateBody()` under a new `"StopFailure"` key, following the exact shape of
the other entries. Silent failure (`2>/dev/null || true`) so it can never wedge
the agent, consistent with every other hook in the overlay.

### Why this reuses everything already built

`tbd notify` already:
- resolves the worktree (`TBD_WORKTREE_ID` env → PWD fallback) and terminal
  (`TBD_TERMINAL_ID` env), exiting silently when unresolvable
  (`Sources/TBDCLI/Commands/NotifyCommand.swift`);
- routes through `RPCMethod.notify` → daemon `handleNotify()` →
  `db.notifications.create(...)` → `StateDelta.notificationReceived` broadcast;
- the app then plays the notification sound and posts a macOS banner, and the
  sidebar shows the unread badge.

The `.error` `NotificationType` already exists and is the highest severity
(`Sources/TBDShared/Models.swift`). No new type, no new RPC, no new DB column,
no app-side change.

## Out of scope

- **Transcript / output scanning, `asyncRewake`** — unnecessary; `StopFailure`
  is push-based and confirmed stable.
- **Codex sessions** — Codex uses a separate hook system. Whether it emits an
  API-error-equivalent event is unresearched; deferred to a follow-up.
- **New notification type / per-error-type wording** — the generic
  `Claude stopped: API error (rate_limit)` message is clear and actionable. A
  dedicated `tbd hooks api-error` subcommand with friendlier per-type messages
  (Option B) can be added later if the generic wording proves too terse (YAGNI).

## Testing

- **Overlay generation test** (the gated-behavior test CLAUDE.md requires):
  assert `ClaudeHookOverlay.generateBody()` produces a `StopFailure` event whose
  command is `stopFailureCommand`. This is the test for the new branch/entry.
- The `tbd notify --type error` path is already covered by existing notification
  machinery; no new coverage needed there.
- A live end-to-end check requires triggering a real API rate-limit error, which
  is not reliably reproducible on demand. We therefore verify the overlay shape
  in tests and confirm the `.error` notification path independently, rather than
  forcing a live error. After restart, `tbd hooks status` should show the
  `StopFailure` event registered in the overlay.
